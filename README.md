## 🔄 End-to-End Request Flow
 
### Flow 1 — Media Analysis (Audio / Video / Image)
 
```
╔══════════════════════════════════════════════════════════════════════════════════╗
║               MEDIA ANALYSIS  —  END-TO-END FLOW                                 ║
╠══════════════════════════════════════════════════════════════════════════════════╣
║                                                                                  ║
║  [1] USER ACTION                                                                 ║
║      User uploads file via file-dropzone / audio-upload / video-upload           ║
║      └── use-media-upload hook validates type & size on client                   ║
║                                                                                  ║
║  [2] API REQUEST                                                                 ║
║      POST /api/v1/analyze/{audio|video|image}                                    ║
║      Headers: Authorization: Bearer <JWT>                                        ║
║      Body: multipart/form-data (file binary)                                     ║
║                                                                                  ║
║  [3] GATEWAY — routes/analyze.py                                                 ║
║      ├── JWT auth check (dependencies.py)                                        ║
║      ├── File type & size validation (utils/file_validation.py)                  ║
║      ├── SHA-256 hash computed (utils/hashing.py)                                ║
║      └── Dispatch to correct service                                             ║
║                                                                                  ║
║  [4] PREPROCESSING                                                               ║
║      Audio  → resample 16kHz → pad/trim 4s → Mel-spectrogram (128,128,1)         ║
║      Video  → extract 1 FPS frames → MediaPipe face detect → crop (128,128,3)    ║
║      Image  → PIL load → MediaPipe face detect → crop (128,128,3)                ║
║                                                                                  ║
║  [5] ML INFERENCE  (ml/audio_model.py | video_model.py)                          ║
║      Model loaded via model_loader.py (cached on startup)                        ║
║      Input tensor → CNN forward pass → Raw probability [0.0 – 1.0]               ║
║      Video: per-frame probabilities → mean aggregation → video score             ║
║                                                                                  ║
║  [6] RULE-BASED HEURISTICS  (services/scoring_engine.py)                         ║
║      Checks run in parallel with ML inference:                                   ║
║      ├── Audio: compression artefacts, silence ratio, clipping detection         ║
║      ├── Video: frame consistency, metadata anomalies, codec fingerprints        ║
║      └── Image: EXIF metadata, noise patterns, edge artefacts                    ║
║                                                                                  ║
║  [7] RISK FUSION  (services/scoring_engine.py)                                   ║
║      final_score = (ml_prob × 0.75) + (rule_score × 0.25)                        ║
║      ├── final_score ≤ 0.30  →  SAFE                                             ║
║      ├── final_score 0.30–0.70  →  SUSPICIOUS                                    ║
║      └── final_score ≥ 0.70  →  DANGEROUS                                        ║
║                                                                                  ║
║  [8] EXPLAINABILITY  (services/explainability.py)                                ║
║      Generates human-readable explanation:                                       ║
║      ├── "Model detected spectral artefacts consistent with synthesis"           ║
║      ├── Triggered rules: ["high silence ratio", "clipping detected"]            ║
║      └── Safe next steps + CERT-In/RBI reference links                           ║
║                                                                                  ║
║  [9] PERSISTENCE                                                                 ║
║      Analysis record saved to PostgreSQL:                                        ║
║      { user_id, file_hash, ml_score, rule_score, final_score,                    ║
║        risk_level, explanation, triggered_rules, created_at }                    ║
║                                                                                  ║
║  [10] RESPONSE  →  Frontend                                                      ║
║       { risk_level, final_score, explanation, triggered_rules,                   ║
║         ml_score, rule_score, safe_next_steps }                                  ║
║       Rendered by: result-card + risk-gauge + explanation-panel                  ║
║                                                                                  ║
╚══════════════════════════════════════════════════════════════════════════════════╝
```
 
### Flow 2 — Phishing / URL / QR Analysis
 
```
╔══════════════════════════════════════════════════════════════════════════════════╗
║               PHISHING / URL / QR  —  END-TO-END FLOW                            ║
╠══════════════════════════════════════════════════════════════════════════════════╣
║                                                                                  ║
║  [1] USER INPUT                                                                  ║
║      ├── Email: paste subject + body via email-form.tsx                          ║
║      ├── URL:   paste link via url-form.tsx                                      ║
║      └── QR:    upload image via qr-upload.tsx                                   ║
║                                                                                  ║
║  [2] API REQUEST                                                                 ║
║      POST /api/v1/analyze/{email|url|qr}                                         ║
║                                                                                  ║
║  [3] TEXT PREPROCESSING  (utils/text_preprocessing.py)                           ║
║      Email → lowercase → strip URLs → tokenize → TF-IDF feature vector           ║
║      URL   → parse components (scheme, hostname, path, params, TLD)              ║
║      QR    → pyzbar decode → extract payload (URL or UPI intent string)          ║
║                                                                                  ║
║  [4] ML INFERENCE  (ml/phish_model.py)                                           ║
║      Email: TF-IDF vector → scikit-learn classifier → phishing probability       ║
║      URL:   lexical feature vector → url_heuristics.py scoring                   ║
║      QR:    decoded payload routed to URL pipeline or UPI checker                ║
║                                                                                  ║
║  [5] HEURISTIC LAYER  (25% weight)                                               ║
║      Email rules:                                                                ║
║      ├── phishing_keywords.json  ("KYC blocked", "urgent action required")       ║
║      ├── upi_scam_patterns.json  (fake collect requests, impersonation)          ║
║      └── Header checks          (From/Reply-To mismatch, spoofed domains)        ║
║                                                                                  ║
║      URL rules:                                                                  ║
║      ├── Lexical: length >75 chars, IP in hostname, excess subdomains            ║
║      ├── Blocklist: url_blocklist.txt matching                                   ║
║      └── India-specific: fake bank/UPI/govt domain patterns                      ║
║                                                                                  ║
║      QR / UPI rules:                                                             ║
║      ├── VPA format validation  (pa=xxx@yyy)                                     ║
║      ├── Merchant name mismatch detection                                        ║
║      ├── Suspicious amount patterns                                              ║
║      └── Known fraudulent VPA matching                                           ║
║                                                                                  ║
║  [6] RISK FUSION + EXPLANATION  →  same as media flow (steps 7–10)               ║
║                                                                                  ║
╚══════════════════════════════════════════════════════════════════════════════════╝
```
